# NIST SP 800-53 Rev5 Control Assessment
**System Name:** FinanceTracker
**Assessment Date:** 2026-04-05
**System Type:** Web Application
**Assessment Scope:** Application-layer controls (infrastructure controls noted as Planned)

---

## AC — Access Control

### AC-1 — Policy and Procedures
| Field | Value |
|-------|-------|
| **Status** | Not Applicable |
| **Implementation** | AC-1 requires organizational policy and procedure documentation. FinanceTracker is a personal-use, single-user finance tracker with no organizational governance structure described. No access control policy document exists in the repository. |
| **Evidence** | No `SECURITY.md`, `ACCESS_POLICY.md`, or equivalent policy artifact found in `personal-finance-tracker` repository. |
| **Gaps** | If deployed in an organizational context, a formal access control policy and associated procedures must be documented. |

---

### AC-2 — Account Management
| Field | Value |
|-------|-------|
| **Status** | Not Implemented |
| **Implementation** | The application has no user account management system. `backend/main.py` contains no authentication middleware, no user registration or login endpoints, and no session or token issuance logic. All API endpoints (`/transactions`, `/summary`, `/export`) are openly accessible without any account context. |
| **Evidence** | `backend/main.py` — no `User` model, no `/auth/register`, no `/auth/login` routes, no `Depends(get_current_user)` dependency injection on any route handler. `frontend/index.html` — no login form or session management UI. |
| **Gaps** | No user accounts exist. The application must implement user registration, authentication, account lifecycle management (creation, modification, disabling, deletion), and role assignment before this control can be satisfied. |

---

### AC-3 — Access Enforcement
| Field | Value |
|-------|-------|
| **Status** | Not Implemented |
| **Implementation** | No access enforcement mechanisms are present at the application layer. All FastAPI route handlers are unauthenticated and unprotected. Any client with network access to the running service can read, create, edit, or delete any transaction and export all financial data without restriction. |
| **Evidence** | `backend/main.py` — route decorators such as `@app.get("/transactions")`, `@app.post("/transactions")`, `@app.delete("/transactions/{id}")`, and `@app.get("/export")` carry no security dependencies (e.g., no `Depends(oauth2_scheme)` or equivalent). |
| **Gaps** | Authentication and authorization must be enforced on every route. Least-privilege access enforcement cannot be implemented until AC-2 (account management) is addressed. |

---

### AC-6 — Least Privilege
| Field | Value |
|-------|-------|
| **Status** | Not Implemented |
| **Implementation** | Because there are no user accounts or roles, least-privilege principles cannot be applied. The SQLite database connection in `backend/main.py` is opened with full read/write access using the default `sqlite3.connect()` call. No read-only roles or scoped database users are defined. |
| **Evidence** | `backend/main.py` — `sqlite3.connect(...)` used for all operations with no differentiation between read and write access paths. No role-based access control (RBAC) structures present. |
| **Gaps** | Define user roles (e.g., viewer vs. editor). Apply database connection scoping. Restrict the CSV export endpoint to authorized users only. |

### AC-7 — Unsuccessful Logon Attempts
| Field | Value |
|-------|-------|
| **Status** | Not Implemented |
| **Implementation** | No authentication layer exists; therefore, no logon attempt tracking, lockout, or throttling mechanism is present. There is no brute-force protection on any endpoint. |
| **Evidence** | `backend/main.py` — no login endpoint, no failed-attempt counter, no account lockout logic, no rate-limiting middleware (e.g., no `slowapi` or equivalent dependency in `backend/requirements.txt`). |
| **Gaps** | Implement authentication first (AC-2/AC-3). Once authentication is in place, add logon attempt counting, configurable lockout thresholds, and delay/lockout enforcement. |

### AC-17 — Remote Access
| Field | Value |
|-------|-------|

No explicit TLS termination is configured at the application layer; TLS would need to be handled by an upstream reverse proxy (not present in the repository). |

## AU — Audit and Accountability

### AU-2 — Event Logging
| Field | Value |
|-------|-------|
| **Status** | Partially Implemented |
| **Implementation** | Python's standard `logging` module is configured in `backend/main.py` with `INFO`-level logging and a timestamped format. The logger `finance_tracker` is instantiated and available throughout the backend. However, logging calls are not systematically applied to all security-relevant events (e.g., transaction deletion, export actions, errors). |
| **Evidence** | `backend/main.py` — `logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(name)s – %(message)s")` and `log = logging.getLogger("finance_tracker")`. No evidence of `log.info(...)` or `log.warning(...)` calls surrounding delete, export, or error-handling code paths in the provided excerpt. |
| **Gaps** | Define a formal set of auditable events (transaction create/edit/delete, export, application errors, startup/shutdown). Ensure `log.info()` / `log.warning()` / `log.error()` calls are present at each event. Log records should include timestamp, event type, resource identifier, and outcome. |

---

### AU-3 — Content of Audit Records
| Field | Value |
|-------|-------|
| **Status** | Partially Implemented |
| **Implementation** | The logging format string `"%(asctime)s %(levelname)s %(name)s – %(message)s"` captures timestamp, severity level, and logger name. This provides a minimal audit record structure. However, without a user identity system, records cannot include user ID, source IP, or session identifiers, which are required AU-3 fields. |
| **Evidence** | `backend/main.py` — `format="%(asctime)s %(levelname)s %(name)s – %(message)s"`. FastAPI's default Uvicorn access log captures HTTP method and path, but user identity context is absent. |
| **Gaps** | Audit records must include: user identity (requires AC-2 implementation), source IP address (can be obtained from `Request.client.host` in FastAPI), event outcome (success/failure), and the specific object acted upon (e.g., transaction ID). |

### AU-9 — Protection of Audit Information
| Field | Value |
|-------|-------|
| **Status** | Planned |
| **Implementation** | Log output is written to `stdout`/`stderr` via `logging.basicConfig`. In a containerized deployment (`docker-compose.yml`), logs are captured by the Docker logging driver. No application-layer mechanisms protect log integrity or restrict log access. Protection of log storage is an infrastructure-layer concern. |
| **Evidence** | `backend/main.py` — `logging.basicConfig(...)` with no file handler or log-forwarding configuration. `docker-compose.yml` — no `logging:` driver configuration specifying a secure log sink. |
| **Gaps** | Configure a centralized, write-protected log sink (e.g., syslog, SIEM). Restrict log access to authorized personnel. Implement log integrity checks (e.g., cryptographic signatures or tamper-evident log shipping). These are primarily infrastructure-layer controls. |

### AU-12 — Audit Record Generation
| Field | Value |
|-------|-------|
| **Status** | Partially Implemented |
| **Implementation** | The logging framework is initialized and capable of generating audit records. FastAPI/Uvicorn automatically logs HTTP requests. | **Gaps** | Add explicit audit-record-generating `log.*` calls within each route handler for create, update, delete, and export operations. Ensure all audit records are timestamped and include sufficient context per AU-3. |

## SC — System and Communications Protection
